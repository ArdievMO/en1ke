1.2. Создание pipeline

Для пайплайн созданы следующие стейджи:
    - checkstyle
    - builde
    - test
    - deploy

В stage checkstyle используется библиотека для проверки стилистики кода. В файл build.gradle добавлена функция:
checkstyle {
    toolVersion = '10.12.1'
    config = rootProject.resources.text.fromUri("https://raw.githubusercontent.com/checkstyle/checkstyle/checkstyle-10.12.1/src/main/resources/google_checks.xml")
}

Для билда необходимо прохождение checkstyle. В этом стейдже собирается сам проект функцией: "./gradlew clean build --stacktrace --info --build-cache"
Так же с указанием флага "expire_in: 1 day" собранные артифакты хранятся  сутки.
Затем замоканы стейджи тестов и деплоя, которые не пройдут без пройденного билда. При запуске пайплайна не прошел первый стейдж.

1.3. Скрипт

На питоне написан скрипт для проверки размера MR:
    - Получает параметры запуска: URL Github, токен доступа и ID проекта, что делает код универсальным для любого проекта;
    - Вычисляет общий размер изменений как разницу между добавленными и удаленными строками в уже завершенных MR;
    - Проверяет ограничения в зависимости от типа ветки:
        - feature — не более 300 строк;
        - bugfix — не более 150 строк;
        - refactor — не более 400 строк;
    - При превышении лимита скрипт завершается с ошибкой;

Для запуска в пайплайне в стейдже чекстайл прописана команда: python main.py $url $gl_token $project_id

Так же создан 2 скрипт для эпик веток.
    - скрипт запускается при MR из фича ветки в эпик
    - в открытый MR эпик ветки добавляет описание: 
        - все feature-ветки, связанные с этим эпиком
        - для каждой фича-ветки находит соответствующий MR
        - получает информацию о задаче из описания MR, а так же закрыт ли MR фича-ветки
